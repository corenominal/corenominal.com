# corenominal.com - Authentication

## Overview

The authentication system is a custom implementation built on CodeIgniter 4. It supports form-based authentication for web users and API key authentication for programmatic access.

---

## Architecture

Authentication operates in two distinct modes:

- **Session-based** — web users authenticate via login form; credentials are stored in a server-side session and optionally persisted via cookies
- **API key-based** — programmatic access is authenticated via HTTP headers; no session is created

---

## Database Tables

| Table | Purpose |
|---|---|
| `users` | User accounts |
| `groups` | Role assignments per user |
| `apikeys` | API keys (auto-generated on login and manually created) |
| `password_resets` | One-time password reset tokens |
| `login_attempts` | Failed login attempts for brute-force protection |

### `users`

| Column | Type | Notes |
|---|---|---|
| `id` | INT PK | |
| `uuid` | VARCHAR | Indexed; used as a stable external identifier |
| `email` | VARCHAR | |
| `username` | VARCHAR | |
| `realname` | VARCHAR | |
| `password` | VARCHAR(255) | bcrypt hash |
| `validated` | TINYINT | 0 = unverified; 1 = verified |
| `banned` | TINYINT | 1 = account is suspended |
| `created_at / updated_at / deleted_at` | DATETIME | Soft deletes enabled |

Login requires `validated = 1` and `banned = 0`.

### `groups`

One row per role assignment. The only current group value is `administrators`. A user can belong to multiple groups.

### `apikeys`

Stores both auto-generated keys (created on each login) and manually created keys. The `autogenerated` flag distinguishes the two. Soft deletes allow keys to be retired without data loss.

### `password_resets`

Stores one-time reset tokens (UUIDs) with a 1-hour TTL. The record is deleted after the password is updated to prevent reuse.

### `login_attempts`

| Column | Type | Notes |
|---|---|---|
| `id` | INT PK | |
| `ip_address` | VARCHAR(45) | IPv4 or IPv6 address of the requester |
| `email` | VARCHAR | Email that was attempted |
| `created_at / updated_at` | DATETIME | |

Records a row for each failed login. Rows are never soft-deleted; they expire naturally as the rolling window query ages them out. All rows for an IP are deleted on successful login.

All authentication routes are grouped under `/auth`.

|---|---|---|---|

## Login Flow

1. User submits email and password via the login form (handled by `login.js` as a `fetch` POST).
2. `Auth\Login::process` checks whether the requester's IP is blocked (see [Brute-Force Protection](#brute-force-protection)).
3. The CSRF token is validated.
4. The `users` table is queried by email with `validated = 1` and `banned = 0`.
5. `password_verify()` checks the submitted password against the stored bcrypt hash.
6. On failure: the attempt is recorded in `login_attempts` and a `401` is returned.
7. On success:
   - All `login_attempts` rows for the IP are deleted.
   - A new API key (UUID) is generated and written to the `apikeys` table with `autogenerated = 1`.
   - Session variables are set: `user_uuid`, `apikey`, `email`, `user_id`, `username`, `realname`, `groups`, `is_admin`.
   - Persistent cookies are set: `user_uuid`, `username`, `email`, `apikey`.
   - Cookie lifetime: 1 year if "remember me" is checked; session-only otherwise.
8. The response contains a redirect URL; `login.js` navigates there on success.

---

## Registration Flow

1. User submits the registration form (`register.js` handles client-side validation and the `fetch` POST).
2. `Auth\Register::process` validates the CSRF token, email format, and password strength.
3. Duplicate email and username checks are performed.
4. Password strength requirements: minimum 12 characters; must include uppercase, lowercase, digit, and special character.
5. The password is hashed with `password_hash(PASSWORD_DEFAULT)` and a UUID is generated for `uuid`.
6. **First user registered** is automatically validated (`validated = 1`) and assigned to the `administrators` group — no email verification required.
7. **Subsequent users** have `validated = 0` and receive a verification email. They cannot log in until they verify.

Registration can be disabled entirely via `App::$allowNewUserRegistration` in [app/Config/App.php](app/Config/App.php).

---

## Email Verification

After a non-first-user registers, they are redirected to `/auth/register/verify`. A verification email is sent via the `Sendmail` library containing a unique link. Clicking the link sets `validated = 1` on the user record.

---

## Password Reset Flow

1. User submits their email at `/auth/password-reset`.
2. `Auth\PasswordReset::request` looks up the email. Whether or not the email exists, the response is identical to prevent email enumeration.
3. If the email exists, a UUID reset token is written to `password_resets` and a reset link is emailed.
4. The link points to `/auth/password-reset/confirm/{uuid}`.
5. `Auth\PasswordReset::confirm` validates the UUID, checks it exists in `password_resets`, and verifies it is less than 1 hour old.
6. The user submits a new password (same strength requirements as registration).
7. `Auth\PasswordReset::update` hashes and saves the new password, then deletes the reset record.

---

## Brute-Force Protection

Failed login attempts are tracked by IP address in the `login_attempts` table. If an IP exceeds the configured threshold within the rolling window, subsequent login requests return `429 Too Many Requests` without checking credentials.

Configuration is in [app/Config/Fail2Ban.php](app/Config/Fail2Ban.php):

| Setting | Default | Description |
|---|---|---|
| `$maxAttempts` | `5` | Failed attempts before the IP is blocked |
| `$banWindowMinutes` | `15` | Rolling window length and effective ban duration (minutes) |

The ban is not permanent — once attempts older than `$banWindowMinutes` age out of the window, the IP can log in again. On a successful login, all `login_attempts` rows for the IP are immediately cleared.

The logic lives in [app/Models/LoginAttemptModel.php](app/Models/LoginAttemptModel.php) and is invoked at the start of `Auth\Login::process`.

## Authorization Filters

Filters are registered in [app/Config/Filters.php](app/Config/Filters.php) and applied to route patterns.

### `SessionFilter` — [app/Filters/SessionFilter.php](app/Filters/SessionFilter.php)

Applied globally (before every request). If the session is not already populated and a valid `apikey` cookie is present, it hydrates the session with user data. Never redirects — public routes remain accessible to unauthenticated users.

### `AdminFilter` — [app/Filters/AdminFilter.php](app/Filters/AdminFilter.php)

Applied to `admin` and `admin/*`. Redirects to `/auth` if the session has no `user_uuid`; redirects to `/unauthorised` if `is_admin` is falsy.

### `ApiFilter` — [app/Filters/ApiFilter.php](app/Filters/ApiFilter.php)

Applied to `api` and `api/*`. Authenticates via HTTP headers; no session is involved.

Two authentication paths:

1. **Master key** — compares the `apikey` header against `config('ApiKeys')->masterKey` (set in `.env`). Sets `$GLOBALS['is_admin'] = true`.
2. **Per-user key** — requires both `apikey` and `user-uuid` headers. Looks up a matching record in `apikeys`. Sets `$GLOBALS['is_admin'] = true` if the record has `is_admin = 1`.

Returns `401 Unauthorized` for missing or invalid credentials.

CORS headers are set to allow all origins with `apikey`, `user-uuid`, and `email` as allowed headers. OPTIONS preflight requests are handled.

---

## API Key Management

- An API key is auto-generated (UUID) on every successful login and stored in `apikeys` with `autogenerated = 1`.
- Manually created keys can also be added to `apikeys` (with a `note` for identification).
- Keys are soft-deleted when retired; they remain in the database but are inactive.

## Security Measures

| Measure | Detail |
|---|---|
| Password hashing | `password_hash(PASSWORD_DEFAULT)` (bcrypt) |
| Password strength | 12+ chars; upper, lower, digit, special char required |
| CSRF protection | Validated on all state-changing POST requests |
| Email enumeration | Password reset returns identical response for known/unknown emails |
| Account gates | Login blocked if `validated = 0` or `banned = 1` |
| Reset token TTL | Password reset tokens expire after 1 hour |
| One-time tokens | Reset token deleted immediately after use |
| Brute-force protection | IP blocked for 15 min after 5 failed attempts (configurable) |
